## STD-REPAIR — Standard Output Repair (Semantic + Contract)

### Input Grammar

**Simple prose (2 required blanks)**
Repair **[Action Output Pack : Targets]** to satisfy **[Action Spec : Targets]**.

**Required blanks (2)**

* **[Action Output Pack : Targets]** — the chunks + patch_segments produced by a standard action run (may be thin/offspec).
* **[Action Spec : Targets]** — the spec pack for that action (What to Produce + Beat Categories + Output Format rules).

**Pro prose (optional boosters)**
Repair **[Action Output Pack : Targets]** to satisfy **[Action Spec : Targets]**, using **[Verifier Notes : Chronicle]** and grounded by **[Run Inputs : Targets]**.

**Optional blanks (descriptions)**

* **[Verifier Notes : Chronicle]** — findings that highlight what’s missing/contradictory/offspec.
* **[Run Inputs : Targets]** — the input packs used for the original run (intent pack, target pack, packet pack, chronicle pack) to keep repairs aligned.

---

### What to Produce

Produce a **Repaired Output Pack** that satisfies the action’s **What to Produce** contract while staying within global chunk/patch budgets.

Repair scope (what you are allowed to change):

1. **Fill missing contract elements** — add minimal new TEXT chunks that cover missing required items.
2. **Strengthen weak elements** — rewrite thin/vague contract parts into concrete statements.
3. **Resolve contradictions** — reconcile conflicting statements by choosing the smallest coherent set that preserves the core aim.
4. **Tighten scope** — if output is overbroad, cut to the minimal slice that still satisfies the contract; move cut items into a single “Deferred: …” line only if needed.
5. **Patch segments (if present)**:

   * keep 1–3 segments max,
   * shrink overly broad segments into smaller local edits,
   * ensure TEXT framing matches CODE,
   * avoid introducing new scope via code changes.
6. **Beat coverage** — ensure the final set broadly covers recommended beats, without naming beats in titles.

Hard guards:

* Do not introduce brand-new objectives not implied by the spec or run inputs.
* Do not invent irrelevant surfaces; if grounding is thin, prefer smaller repairs and explicit “Assumed: …” in TEXT (no questions).
* Do not add more chunks than necessary; prioritize minimal completeness.
* No internal action IDs and no “next move” guidance.

---

### Recommended Chunk Beat Categories (with 1-line descriptions)

* **Contract Completion** — missing required elements added in minimal form.
* **Clarity Upgrade** — vague statements rewritten into concrete, testable language.
* **Contradiction Reconciliation** — conflicts removed while preserving core aim.
* **Scope Tightening** — output reduced to a winnable slice without losing contract coverage.
* **Patch Repair (optional)** — patch segments reduced and localized to fit budgets and intent.
* **Beat Coverage Alignment** — composition adjusted to cover the intended shape without beat-title leakage.
