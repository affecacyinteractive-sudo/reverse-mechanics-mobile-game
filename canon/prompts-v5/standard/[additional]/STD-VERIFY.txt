## STD-VERIFY — Standard Action Output Verifier (Universal)

### Input Grammar

**Simple prose (2 required blanks)**
Verify **[Action Output Pack : Targets]** against **[Action Spec : Targets]**.

**Required blanks (2)**

* **[Action Output Pack : Targets]** — the produced chunks + patch_segments from one standard action run (the thing being judged).
* **[Action Spec : Targets]** — the spec for that action (What to Produce + Beat Categories + Output Format rules as a target pack).

**Pro prose (optional boosters)**
Verify **[Action Output Pack : Targets]** against **[Action Spec : Targets]**, grounded by **[Run Context : Chronicle]** and constrained by **[Verification Intent : Intent]**.

**Optional blanks (descriptions)**

* **[Run Context : Chronicle]** — any notes about what this run was trying to do (intent pack, target pack selection, known constraints).
* **[Verification Intent : Intent]** — what to prioritize in verification (contract completeness, patch discipline, chunking quality, grounding).

---

### What to Produce

Produce a **Verifier Result Pack** (TEXT chunks) that diagnoses quality without rewriting the output and without “next move” routing.

Across the pack, include:

1. **Verdict (TEXT)** — choose exactly one:
   **OK / THIN / OFFSPEC / FORMAT_FAIL / PATCH_RISK / GROUNDEDNESS_RISK**

2. **Top Findings (TEXT)** — 3–8 short findings, each written as:
   `Finding: <problem> → Evidence: <where it shows up> → Impact: <what it breaks>`

3. **Contract Coverage (TEXT)** — a compact checklist summary:

   * which required contract elements are present
   * which are missing or too weak
     (No long tables; keep it readable.)

4. **Chunking & Title Quality (TEXT)** — one tight summary of:

   * title rule compliance (≤25 chars, not beat names, grounded)
   * chunk size discipline (word/line budgets, one-idea-ness)
   * ordering/coverage (beats covered without copying beat names)

5. **Patch Segment Quality (optional, TEXT)** — only if patch_segments exist:

   * locality (small bounded edits)
   * framing (TEXT matches CODE)
   * budget (1–3 segments, 8–12 code lines)
   * risk notes (rewrite smell, too broad)

6. **Grounding Note (optional, TEXT)** — only if the output references surfaces that appear unsupported by provided inputs/spec targets; keep it as “risk” not a blocker.

Hard guards:

* Do not edit or regenerate the action output here.
* Do not reference internal action IDs or recommend “next best move.”
* Do not ask questions.

---

### Recommended Chunk Beat Categories (with 1-line descriptions)

* **Verdict** — the single readiness label for this output pack.
* **Top Findings** — the most important failures, each with evidence + impact.
* **Contract Coverage** — which contract elements are satisfied vs missing/thin.
* **Chunking & Title Quality** — whether composition rules were followed cleanly.
* **Patch Segment Quality (optional)** — only when code patches exist; checks patch discipline.
* **Grounding Note (optional)** — flags unsupported references as a risk signal.
