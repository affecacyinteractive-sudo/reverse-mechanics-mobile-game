## Deterministic Heuristics (V1)

### A) Preprocessing (deterministic)

Compute from the raw message text:

* `char_count` = number of non-whitespace characters
* `word_count` = whitespace token count
* `line_count` = number of lines after trimming
* `sent_count` (approx) = count of `. ! ?` (clamp to [1..])
* `has_code` = true if contains any of: ``` fences, `;{}` density, common keywords (`function`, `class`, `import`, `SELECT`, `def`, etc.)
* `has_numbers` = true if contains digits
* `has_quotes` = true if contains quoted phrases `"..."` or `'...'`

### B) `Too Short` (quantity threshold)

Trigger `Too Short` if **any** of these are true:

* `char_count < 60` **OR**
* `word_count < 12` **OR**
* `sent_count < 1` *(empty/near-empty after trim)*

Severity upgrade (still same seal title) if:

* `char_count < 30` OR `word_count < 6`
  Body should say “very short; likely underspecified.”

### C) `No Concrete Subject` (specificity threshold)

Goal: detect whether the message names at least one concrete thing it’s about.

Trigger `No Concrete Subject` if **all** of these are true:

* No quoted phrase AND
* No file/path-ish token AND
* No identifier-ish token AND
* No UI/object noun signal

Define signals (any one is enough to avoid the flag):

1. **File/path signal**

* contains `/` or `\` with letters around it, or a filename extension like `.ts .tsx .js .py .java .cs .go .rs .sql .yaml .yml .json`

2. **Identifier-ish signal**

* contains camelCase / PascalCase / snake_case token length ≥ 6
  (regex style: `[a-z]+[A-Z][A-Za-z0-9]*` OR `[A-Z][a-z]+[A-Za-z0-9]*` OR `[a-z]+_[a-z0-9_]+`)

3. **UI/object noun signal** *(simple lexicon)*

* message contains at least one noun-like anchor from a small maintained list, e.g.:
  `button, form, modal, page, screen, table, list, filter, search, login, auth, upload, download, payment, settings, profile, notification, dashboard, report, export, import, sync, cache, database, query, endpoint, api, service, job, pipeline`
  (You can tune this list; it’s deterministic.)

4. **Quoted phrase signal**

* any `"..."` / `'...'` phrase length ≥ 3 words counts as a subject anchor.

If none of those signals exist, the message is likely “do it better / fix it / improve it” without naming what.

### D) `No Done Signal` (missing success condition)

Trigger `No Done Signal` when the message contains **no success cue** and **no acceptance cue**.

Success cues (any one avoids the flag):

* contains one of: `so that`, `so I can`, `until`, `when`, `done`, `works`, `working`, `fixed`, `passes`, `no longer`, `should`, `expected`, `must`, `ensure`, `confirm`, `verify`
* contains comparison structure: `instead of`, `rather than`, `expected vs`, `before/after`
* contains an explicit observable: `shows`, `returns`, `renders`, `saves`, `loads`, `submits`, `creates`, `deletes`, `updates`, `fails`, `throws`

If none are present, the ask is probably aspiration-only (“make it better / add feature”) without a finish line.

### E) `Overbroad Scope` (too big for 1–3 actions)

Trigger `Overbroad Scope` if **any** of these are true:

1. **Multi-object explosion**

* contains 3+ conjunction-separated asks (heuristic): count occurrences of `and`/`,`/`;` where each side has ≥ 4 words.

  * Example: “build X and Y and Z” → overbroad.

2. **Large-system keywords without bounds**

* contains keywords like: `entire`, `whole`, `complete`, `full`, `end-to-end`, `architecture`, `system`, `platform`, `rewrite`, `refactor everything`, `production-ready`
  AND lacks any limiting cue like: `one`, `single`, `minimal`, `small`, `only`, `just`, `one screen`, `one endpoint`, `one flow`.

3. **Time/effort mismatch**

* contains “quick” / “fast” / “in 5 minutes” **and** also contains large-system keywords above.

4. **High-level + no anchor**

* message triggers `No Concrete Subject` **and** word_count > 40
  (This indicates large vague scope.)

Seal body should say: “scope likely exceeds 1–3 actions; needs slicing.”

### F) `Contradictory Ask` (only trivial deterministic contradictions)

Trigger only for obvious pattern pairs (keep this conservative):

* Contains both `no X` and `add X` for same token X within 30 words.
  Example: “don’t use cache… add caching”
* Contains both `must be minimal` and `cover everything` / `full` / `complete`.

If triggered, body should say: “contains a direct contradiction; intent needs reconciliation.”

### G) `Target Needed` (conditional; requires action context)

Only evaluate this if the orchestrator already knows the chosen downstream action.
Trigger if:

* chosen action has `requires_targets = true`
* and `Targets[]` is empty (or no target packs selected)

Seal body: “this action needs at least one target pack to stay grounded.”

### H) Output cap rule (to keep seals small)

* Emit at most **2 seals** by default.
* If more than 2 flags trigger, choose the top 2 by priority:

  1. `Overbroad Scope`
  2. `No Concrete Subject`
  3. `No Done Signal`
  4. `Too Short`
  5. `Contradictory Ask`
  6. `Target Needed` *(only if action is already chosen)*

## Deterministic trimming heuristics (V1)

### A) Hard caps (defaults)

* Max total Packet Pack chunks: **6**
* Max CODE extracts (kind=CODE) inside the pack: **4**

  * 1 Primary + up to 3 Context
* Max lines per CODE extract: **12** (align with global code chunk rule)
* Max TEXT chunks: **3**

  * Summary + (optional) Handles + (optional) Assumptions

### B) Priority order for keeping content

Keep in this order:

1. Primary Surface Extract (must keep)
2. Minimal Imports/Types needed to read the primary
3. One Callsite (if it clarifies invocation)
4. One Governing Config/Constant (if it directly affects behavior)
5. One Neighbor Helper (only if directly called by primary)
6. Handle List (only if there is room)
7. Assumptions (only if needed)

### C) How to trim

* If over cap on CODE extracts: drop lowest priority context extracts first.
* If a CODE extract exceeds 12 lines:

  * trim to the tightest contiguous region that still includes the referenced identifiers
  * prefer removing surrounding whitespace/comments and unrelated branches
* If handle list is long: keep top **8–12** identifiers only.
* If summary is long: cap to **2–5 lines** (already implied by PACK-01/02).

### D) Failure condition

Emit `Slice Budget Failed` only if:

* the primary extract alone cannot be made coherent within 12 lines **and**
* dropping all context still doesn’t fit the budget posture.

## Deterministic resolution rules (V1)

### A) Address types (ordered by reliability)

1. **File Path** (strongest)
2. **Region Key** (if supported; e.g., sentinel/entity keys)
3. **Handle / Identifier Match** (fallback; function/class/component name, selector, route, table)
4. **Keyword Search Match** (last resort; index handles/summaries)

### B) Extraction state

* Use the run/state implied by the Target Pack if present.
* Otherwise default to “latest” state (or the currently active run base state).

### C) How to resolve from a Target Pack

Target Packs may contain (any mix):

* explicit file paths
* entity keys / region identifiers
* handle lists
* summaries containing recognizable identifiers

Resolution procedure:

1. **Harvest explicit file paths**

   * If target pack includes file paths, accept them directly if they exist in index at the chosen state.
2. **Harvest explicit region keys**

   * If region keys exist and index supports them, accept them and attach their file paths.
3. **Harvest handles/identifiers**

   * Collect handles (e.g., `Hero`, `useAuth`, `#hero`, `POST /login`, `users_by_id`).
   * Match against index handles to propose candidate file paths.
4. **Rank candidates**

   * Prefer candidates where:

     * handle appears in the file’s handle list, AND
     * file’s summary includes similar terms, AND
     * file is “recent” (latest_run_id close to state) only as a tie-breaker.
5. **Cap & dedupe**

   * Default cap: max 5 file paths resolved per call (unless intent explicitly says “broad slice”).
   * Dedupe file paths; keep highest-confidence matches.

### D) Ambiguity handling (no questions)

If multiple matches tie and you must choose:

* choose the smallest set that stays coherent:

  * prefer 1–2 file paths over 5 scattered ones
* emit a seal chunk:

  * Title: `Ambiguous Target`
  * Body: one sentence naming the ambiguity (“multiple files matched handle X; chose top 2 by handle density”).

### E) Failure condition

Emit `Target Resolution Failed` if:

* no file paths could be resolved AND
* no region keys could be resolved AND
* no handle matches produce candidates.

### QA/Repair symmetry plan (keep it small)

Use **existing generic QA tools** as the default; only add new ones when a new artifact type genuinely differs.

## 1) Reuse, don’t multiply

* **Packet-side outputs** from the new plan actions should be validated/repaired with:

  * **PACK-VERIFY / PACK-NORM / PACK-REPAIR**
* **Chronicle-side outputs** from the new plan actions should be validated/repaired with:

  * **CHR-VERIFY / CHR-NORM / CHR-REPAIR**
* **Standard action outputs** stay on:

  * **STD-VERIFY / STD-NORM / STD-REPAIR**

## 2) Where each new action fits

* **Aggregate Plan**

  * Use **PLAN-VERIFY / PLAN-REPAIR** (plan is its own contract; keep it separate).
  * Optionally run **STD-NORM**-style formatting rules, but don’t add PLAN-NORM unless needed.

* **Code progress vs plan**

  * Output is a “packet-style report” → **PACK-VERIFY/NORM/REPAIR**
  * Keep **PACK-PROG-REPAIR** only as a convenience; it can be dropped if PACK-REPAIR works well.

* **Journey progress vs plan**

  * Output is a chronicle-style report → **CHR-VERIFY/NORM/REPAIR**
  * Keep **CHR-PROG-REPAIR** only if CHR-REPAIR feels too broad; otherwise drop it.

* **Placement confirms**

  * **PACK-CONF-01** outputs packet-style receipts → **PACK-VERIFY/NORM/REPAIR**
  * **CHR-CONF-01** outputs chronicle-style receipts → **CHR-VERIFY/NORM/REPAIR**

* **Post-intent outcome check**

  * Output is a packet-like evaluation tied to code slices → **PACK-VERIFY/NORM/REPAIR**
  * Keep **POST-INTENT-REPAIR** optional; it’s basically a focused PACK-REPAIR.

## 3) Specialist verifiers only when recurring pain justifies them

Keep only these specialists “on standby”:

* **PACK-PATCH-VERIFY** (only if commit patches are frequently unsafe)
* (Optional later) **PLAN-VERIFY** already exists; it’s justified because plan has a unique structure.

Everything else should be caught by the 3 universal sets:

* STD-*, PACK-*, CHR-*.

## 4) Deterministic gates stay structural and few

* Reuse the existing deterministic gates:

  * **SYS-STD-*** for standard outputs
  * **SYS-PACK-GATE-*** for packet outputs
  * Chronicle usually doesn’t need gates beyond budget shaping unless you see constant overlong entries.

This gives you symmetry without a combinatorial explosion.
