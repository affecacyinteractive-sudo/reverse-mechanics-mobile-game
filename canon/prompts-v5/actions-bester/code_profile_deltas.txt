**Profile Delta blocks (V1)**

---

## P0 — DEFAULT_APP_CODE (delta)

* Prefer **plain, readable control flow**; avoid framework-specific patterns unless grounded.
* Keep function boundaries tight: **one function or one small branch**.
* If naming is unclear, use **neutral placeholder names** that won’t conflict (e.g., `nextState`, `result`, `input`), and keep them consistent within the snippet.
* Avoid introducing new dependencies/imports unless required by the patch.

---

## P1 — WEB_UI_REACT (delta)

* Snippet should be either:

  * a **component fragment** (state, effect, handler, render change), or
  * a **hook/handler fragment**.
* Prefer **minimal React state wiring** (e.g., `useState`, derived values, event handlers). Avoid large refactors.
* Do not introduce new UI libraries unless already present in Packets/Targets.
* Keep JSX minimal: **touch only the necessary lines** to reflect the change.
* If the action implies a seam, prefer **prop-level or callback-level seams** over global state changes.

---

## P2 — SERVER_NODE (delta)

* Snippet should be either:

  * a **route/controller handler fragment**, or
  * a **service function fragment**.
* Prefer explicit input parsing + early returns; avoid complex middleware rewrites.
* If returning errors, use the project’s existing style if visible; otherwise:

  * return a minimal error object / status handling in-place.
* Avoid adding new endpoints/routes unless explicitly required by the action output contract.

---

## P3 — DATA_SQL (delta)

* Emit **only SQL** (no ORM wrapper code) unless Packets clearly show a query-builder style.
* Prefer a single cohesive query: one `SELECT` or one `UPDATE/INSERT` (CTE allowed if it fits in 8–12 lines).
* Use **explicit column lists** when possible; avoid `SELECT *` unless necessary to keep within line limits.
* Keep naming consistent (`t`, `u`, `p` aliases) and avoid long alias chains.
* If schema/table names are unclear, use **neutral placeholders** (`table_name`, `user_id`) and keep them consistent.

---

## P4 — CONFIG_STRUCTURED (delta)

* The CODE snippet must be **valid JSON or valid YAML** (whichever applies).
* Touch the **smallest possible section**; do not paste full files.
* Preserve existing key ordering/style if visible; otherwise keep a conventional order (top-level keys first).
* Avoid comments in JSON. YAML comments only if already used in Packets.
* If the change requires multiple keys, keep them **adjacent** in the snippet to show the cohesive change.

---

## P5 — INFRA_DEVOPS (delta)

* Keep to the smallest deployable unit:

  * Dockerfile: one stage/command adjustment
  * k8s YAML: one resource fragment (`env`, `ports`, `resources`, etc.)
  * CI workflow: one job/step fragment
  * Terraform: one resource/attribute fragment
* Must be syntactically valid for that file type (YAML/HCL/Dockerfile).
* Avoid introducing new resources/services unless explicitly required.
* Prefer additive safe changes (env var, readiness probe tweak, resource limits) over restructuring.

---

## P6 — SCRIPTS_AUTOMATION (delta)

* Snippet should be one cohesive script fragment:

  * Node script: one loop/function/IO section
  * Bash: one small sequence (set vars → run command → handle error)
* Prefer **deterministic behavior**: explicit paths, explicit exit codes.
* Don’t add interactive prompts.
* Keep logging minimal and useful (one or two lines).

---

## P7 — TESTS_QA (delta)

* Snippet should be one test case or one small helper fragment.
* Prefer `describe/it` structure for unit tests; for e2e use a single scenario step fragment.
* Assertions must be specific and tied to the behavior the action claims.
* Avoid adding flaky waits; prefer deterministic signals (selectors/events/responses) if visible.

---

### How these get used

* The **Profile Selector** outputs `segment_profiles[]`.
* The generative action uses **the matching delta block** for each patch segment it emits.
* Everything else (chunk sizing, pairing, titles) remains governed by your global prompt baseline.
