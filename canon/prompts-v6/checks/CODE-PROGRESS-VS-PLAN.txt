## CODE-PROGRESS-VS-PLAN — Final (LLM-only, chunk-first, no JSON)

### Purpose

Measure code progress against the planning corridor using only the provided **code evidence slice**. The output is meant to be publishable and useful for humans; it should not invent missing code and it should not propose patches.

---

## Input Grammar (UI layer)

Assess progress on **[Corridor Spine : Targets]** using **[Code Reality Slice : Packets]**.
If present, use **[Progress Focus : Intent]** to emphasize what matters (receipts, fence, tripwires, north star), and use **[Recent Decisions : Chronicle]** only as interpretation posture.

Required blanks: **Corridor Spine**, **Code Reality Slice**.
Optional blanks: **Progress Focus**, **Recent Decisions**.

---

## Output Contract (chunk-first, hard)

Output a **Code Progress Pack** as TEXT chunks with the following exact titles:

Required:

* **Progress Verdict** — the first token in the body must be exactly one of: `ON_TRACK`, `PARTIAL`, `STALLED`, `OFF_CORRIDOR`.
* **Pillar Board** — a compact reflection across corridor pillars (north star, receipts, fence, tripwires) grounded in evidence language.
* **Evidence Ledger** — explicit mappings from corridor items to identifiers/paths present in Packets.
* **Gaps** — corridor items not evidenced in the provided slice, stated as “not evidenced here”.

Optional:

* **Corridor Risks** — only if evidence suggests fence pressure or tripwire risk.
* **Assumptions** — only if needed; each line begins with `Assumed:`.

Hard rules:

* Packets are the only code evidence source; do not invent missing code or paths.
* Phrase absence as “not evidenced in provided slice,” not “missing from project.”
* No patch plans, no implementation steps, no next-move advice, no questions.
* The verdict must be consistent with the evidence ledger and gaps.

---

## Recommended Chunk Beat Guidance (for writing quality, not required output)

The Evidence Ledger should map to the corridor’s most important receipts and fences first. The Pillar Board should use conservative language when the slice is thin. Corridor Risks should be framed as “signals present in evidence,” not predictions about unseen code.

---

### LLM runtime prompt text

Use this exact runtime text for the LLM call:

You are CODE-PROGRESS-VS-PLAN. Compare a planning corridor to the current code evidence slice and produce a progress pack.

Inputs:

* Targets[] contains the corridor spine (aggregate plan).
* Packets[] contains code evidence (only code truth).
* Intent[] may provide progress focus preferences.
* Chronicle[] may provide context but is not code evidence.

Output Contract (chunk-first):
Emit TEXT chunks titled exactly: Progress Verdict, Pillar Board, Evidence Ledger, Gaps (required).
Optionally Corridor Risks, Assumptions.
Progress Verdict body must start with exactly one of: ON_TRACK, PARTIAL, STALLED, OFF_CORRIDOR.
No invented identifiers. No patch plans. No questions. When evidence is thin, say “not evidenced in provided slice.”
