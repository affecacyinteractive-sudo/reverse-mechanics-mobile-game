## Blank Glossary (V1) — De-blur Pairs

- **Slice Identity** = the **noun label** (“Saved Filters”).  
  **Slice Promise** = the **verb capability** (“save + reapply filters”).

- **Minimum Data/State** = the **one new thing you introduce** to make the slice real (field / event / store / DOM handle / permission bit).

- **Evidence Hook** = why you’re patching *this* surface.  
  **Minimal Cause** = what you think is actually breaking.

- **Minimal New Seam** = the **new hook point** (where the capability plugs in).  
  **Local Surface Change** = the **one place the user/flow will notice it**.

- **Responsibility Transfer (Seam)** = one sentence: “move X from A → B.”  
  **Boundary Shift** = the old→new **ownership claim** (“A no longer owns X; B does”).

We’ll patch around **`[Local Surface / File / Component / Function]`**, guided by **[Evidence Hook]**; our best cause hypothesis is **[Minimal Cause]** (even if it’s just a first guess).

**Line 3 — HOW we change it + WHEN we stop**
Apply `[Patch Segment(s)]` (small edits), keep `[Non-Regression Guard]` in mind, then stop once `[Closure Check]` is true (i.e., re-running the trigger now produces the expected behavior). ## FP-02 — Input Grammar (Liberal)

*(Done Receipt / “acceptance reality”)*

**Line 1 — WHAT + WHERE + WHY**
For **[Subject / slice]**, define what “done” *means* as an **observable receipt**—so completion is unambiguous and judgeable (not vibes). Use **[Directive]** to set the intent, and **[Targets / Blueprint]** only if they help keep the receipt consistent. 

**Line 2 — HOW (the receipt itself: outside + inside)**
State **[User-visible Receipt]** (2–4 crisp “user can now do/see” statements) and **[Data-level Reality]** (1–3 “system reliably makes true” facts). Pull details from **[Projection Slice] + [Context]** if you have them; otherwise drop placeholders/guesses and refine later. 

**Line 3 — WHEN we call it done (bar + exclusions + decision test)**
Add **[Quality Bar]** (1–2 expectations), **[Exclusions Boundary]** (what “done” does *not* include), and a **[Closure Check]** sentence that lets you decide whether a proposed change moves *toward* or *away from* done. Any extra cards you include are just “ingredients” to sharpen wording. 

## FP-08 — Input Grammar (Liberal)

*(Tripwires / early failure alarms)*

**Line 1 — WHY + WHAT we’re protecting**
For **[Subject / slice]**, define **Tripwires** that warn us early when we’re drifting into bad territory (complexity, scope creep, coupling, or quality rot). Use **[Directive]** to say what you’re afraid of; use **[Targets/Blueprint]** if they clarify what “bad” looks like.

**Line 2 — WHEN it’s “going wrong” (signals you can notice)**
List **[Tripwire Signals]** as concrete “if you see X, that’s a red flag” statements—draw from any mix of **projection_slice + context + past pain + guesses**. Group them if you want: **drift signals**, **complexity signals**, **dependency/coupling signals**.

**Line 3 — WHAT we do when a tripwire fires (recovery posture)**
For each signal (or for the set), attach a **[Recovery Move]** like “stop and fence scope,” “extract a seam,” “write a probe,” “add an invariant,” or “re-state done.” Call it done when you have at least **[N]** tripwires that would actually change your behavior if they fired.

## FP-10 — Input Grammar (Liberal)

*(Scope Fence / what we will not absorb)*

**Line 1 — WHY + WHAT we’re fencing**
For **[Subject / slice]**, define a **Scope Fence** so the work stays coherent: what we’re *building now*, and what we’re explicitly refusing to absorb (even if it’s tempting). Use **[Directive]** for intent and urgency; use any targets/blueprint notes as extra flavor.

**Line 2 — WHERE the boundary sits (in vs out)**
Write **[In-Scope Core]** as 2–4 statements (“we will handle…”), then **[Out-of-Scope Exclusions]** as 3–7 statements (“we will not…”). You can pull these from projection/context—or invent and revise later.

**Line 3 — WHEN we refuse (rules + edge posture)**
Add **[Refusal Rules]** (how we say no) and **[Edge Handling Posture]** (what to do when reality hits the fence): defer, route to a new slice, log a future card, or create a dedicated extension. Close with a one-line **Closure Check**: “if it violates the fence, it’s not part of this run.”
